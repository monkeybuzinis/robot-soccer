To successfully build and deploy your autonomous soccer robot following your professor's criteria, you should implement the project in four distinct developmental stages. This "MVP-First" (Minimum Viable Product) roadmap ensures you test the hardware and low-level math before layer-stacking the complex pathfinding and behavior trees.
Here is the complete project blueprint, organized by stage, detailing exactly which files are active, what happens in that stage, and how to verify it works.
Stage 1: Hardware Drivers, Data Parsing & Fundamental Math
Objective: Establish basic communication with the external sensors and set up the mathematical groundwork required for spatial geometry.
* Files to Attach / Keep Active:
   * robotpu-i2c-cam.ts (Camera protocol handler)
   * robotpu-utils.ts (Shared math engine)
* What Happens: * The micro:bit opens communication channels over the I2C multiplexer bus to the ESP32 camera.
   * It starts pulling 18-byte data packets and decodes raw 16-bit millimeter positions for targeted colors (orange ball, white goal paper).
   * robotpu-utils.ts provides clamping and radian/degree conversions.
* How to Verify: Open your serial monitor. You should see steady printouts mapping x_mm and y_mm changes when you wave the orange ball in front of the camera lens.
Stage 2: Odometry Tracking & Active Sensor Searching
Objective: Enable the robot to keep track of its own physical positioning in space and actively search for objects when they disappear from sight.
* Files to Add to the Project:
   * robotpu-odometry (1).ts (SE(2) matrix step tracker)
   * robotpu-search-soccer.js (Head camera servo pan/tilt routine)
* What Happens:
   * Every time the bipedal leg triggers a step action, leftStep() or rightStep() calculates relative motion updates to track coordinate displacement ($x, y, \theta$).
   * If the data packet shows an target object is invalid (lost tracking), a state transition hands control over to the head servo loop to sweep looking for the target.
* How to Verify: Walk the robot forward 3 steps manually. Verify that the odometry matrix accurately changes its physical coordinate values without freezing or throwing errors.
Stage 3: Map Building, Filtering & Basic Approach (The "MVP" Grade B/B+)
Objective: Compensate for data latency, smooth out reading errors, calculate where to stand to score, and drive there.
* Files to Add to the Project:
   * robotpu-localmap.ts (Ground plane grid projection)
   * robotpu-kalman-filter.ts (Constant-velocity tracking estimation)
   * robotpu-viewpoint (1).ts (Arc steering motion control)
   * robotpu-soccer-mvp.ts (Core logic calculator)
* What Happens:
   * The raw camera data coordinates are projected down onto a flat local ground plane map.
   * The 2D Kalman filters smooth out rapid twitches or dropped frames if the camera shakes while walking.
   * soccer-mvp computes a vector line from the center of the goal, through the ball, and places a target "Kick Pose" coordinate $15\text{ cm}$ directly behind the ball.
   * The viewpoint controller reads this pose and outputs walking speeds to steer the robot in a sweeping arc so it automatically squares up to face the goal.
* How to Verify: The robot should track the ball, walk around it in a smooth curve, stand directly behind it facing the goal, and stop.
Stage 4: Advanced Obstacle Avoidance & The Behavior Tree (The "A/A+" Grade)
Objective: Prevent collisions with field obstacles, manage logical decisions cleanly, and execute the physical kick.
* Files to Add to the Project:
   * robotpu-A-star (1).ts (Grid pathfinding)
   * robotpu-pure-pursuit (1).ts (Multi-waypoint tracking)
   * robotpu-soccer-final (1).ts (The final integrated Behavior Tree script)
* What Happens:
   * Instead of driving blindly towards the kick point, the local ground plane maps out grid obstacles (like opponent robots).
   * A-star finds the shortest cell pathway around the obstacles.
   * pure-pursuit steps through those path waypoints sequentially, feeding lookahead coordinates directly into the movement system.
   * The Behavior Tree structured loop systematically evaluates condition nodes (condBallVisible, condScored) to seamlessly toggle between searching, chasing, and kicking without massive nested code statements.
* How to Verify: The robot navigates around obstacles, stops precisely when it hits the arrival threshold behind the ball, triggers its striking macro to score, and flashes a celebration icon.
⚠️ Critical Compilation Step for the Final Deployment
Once you reach Stage 4, you must merge the logic blocks into a single workspace.
Because separate files in MakeCode share a global namespace, keeping all these files separate inside your folder simultaneously will cause "Duplicate Identifier" compiler crashes due to overlapping definitions of constants like SIZE, variables like walkSpeed, or classes like Pose2D. Your robotpu-soccer-final (1).ts file is already structurally packed to contain these variables together—ensure it is the lone master file in your final compiled deployment!
